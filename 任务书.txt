AI女友 + 桌宠 项目改造任务书

项目代号：LightAgent-Live
目标效果：Grok Aurora级别的AI女友 + 桌宠悬浮窗
核心原则：简单优先、单一职责、能跑通再扩展

一、项目目标（Spec）
最终要实现的两个场景
场景1：App内AI女友
用户打开App → 看到Live2D角色在屏幕上
用户说话/打字 → AI回复
AI回复时 → 角色嘴巴动（口型同步）
             角色有表情变化（开心/害羞/思考）
             角色有动作（点头/挥手）
             本地语音朗读回复内容

场景2：桌宠模式
用户点击最小化 → 角色缩小悬浮在屏幕角落
退出App后 → 角色还在，悬浮在其他App上方
点击悬浮角色 → 弹出聊天界面
长按悬浮角色 → 可以拖动位置
二、技术选型（决策类Spec）
已确定，不允许推翻的决策
模块	选型	理由
Live2D渲染	WebView + pixi-live2d-display	无需NDK，纯Kotlin可控
本地TTS/ASR	sherpa-onnx	离线、Apache 2.0、支持中文
桌宠悬浮窗	Android FloatingWindowService	系统原生，最稳定
情感分析	本地关键词匹配	够用，零延迟，不依赖网络
模型文件	Live2D官方免费示例（Hiyori）	可立即使用，无版权问题
明确不做的事（边界）
❌ 不接 Live2D Cubism Native SDK（NDK太复杂）
❌ 不做3D模型（超出当前范围）
❌ 不做自定义模型生成（ai_game_workbench是另一个项目）
❌ 不改现有聊天逻辑（只在回复后增加联动）
❌ 不引入新的网络请求库（项目已有的够用）
三、模块拆分
每个模块职责一句话说清楚
Live2DWebView      → 负责渲染Live2D角色，仅渲染，不管业务
SherpaOnnxTTS      → 负责文字转语音+输出PCM音量数据，不管UI
SherpaOnnxASR      → 负责麦克风输入转文字，不管后续处理
EmotionAnalyzer    → 负责文本→情感映射，纯函数，无副作用
FloatingPetService → 负责桌宠悬浮窗生命周期，不管AI逻辑
ChatViewModel      → 负责串联所有模块，是唯一的调度中心
四、开发任务清单
第一阶段：让角色显示出来（优先级最高）

做什么：下载官方Hiyori模型，放入assets/live2d/hiyori/

做什么：写一个H5页面，用pixi-live2d-display加载模型
验收标准：
  - WebView打开后能看到Hiyori站在屏幕上
  - 不报JS错误
  - 背景透明

Task 1.4 — 实现 CubismWebLive2DController.kt

验收标准：

  - 替换掉NoOpLive2DController
负责人：我来写
第二阶段：让角色会说话（口型同步）

Task 2.1 — 集成 sherpa-onnx TTS

做什么：实现SherpaOnnxTTSController，替换NoOpTTSController
验收标准：
  - AI回复后能听到语音朗读
  - 朗读时能拿到实时音量值（0.0~1.0）
负责人：我来写

Task 2.2 — 口型同步联动

做什么：TTS音量值 → setLipSync() → 角色嘴巴动
验收标准：
  - 说话时嘴巴张合
  - 停止说话后嘴巴闭合
  - 口型和声音基本同步（误差<100ms）
负责人：我来写
第三阶段：让角色有情绪

Task 3.1 — 实现 EmotionAnalyzer

做什么：分析AI回复文本，输出情感标签
验收标准：
  - 输入"哈哈好开心" → 输出 happy
  - 输入"让我想想" → 输出 thinking
  - 输入"讨厌啦" → 输出 shy
  - 纯函数，无网络请求，无副作用
负责人：我来写

Task 3.2 — 情感驱动表情+动作

做什么：EmotionAnalyzer结果 → setExpression + playMotion
验收标准：
  - AI回复后角色表情随文字内容变化
  - 不同情感对应不同动作组
负责人：我来写
第四阶段：桌宠悬浮窗

验收标准：

  - 角色悬浮在所有App上方
  - 可拖动位置
  - 点击弹出聊天界面
  - App退出后依然存在

Task 4.2 — App内最小化按钮

验收标准：
